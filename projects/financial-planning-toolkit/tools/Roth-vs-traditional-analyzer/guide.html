Service Guide
Roth vs Traditional – Real‑Market Engine
This guide explains how the tool behaves, how modes are selected, how inputs are interpreted, and how guardrails protect users from confusing or invalid configurations. It doubles as both UX documentation and a mental model for anyone maintaining or extending the engine.
01 · Mental Model How the tool thinks
Core idea
The tool compares Roth vs Traditional outcomes using either real‑market data or a manual growth rate, depending on how the user configures the inputs. The user experience is built around three explicit modes:
  • Real‑Market (Single Ticker): One ticker in the Ticker field, Portfolio empty. Uses historical CAGR and volatility for that asset.
  • Real‑Market (Portfolio): Multiple tickers with weights in the Portfolio field, Ticker empty. Uses weighted portfolio CAGR and volatility (with correlations).
  • Synthetic (Manual Growth): No ticker and no portfolio, or real‑market data unavailable. Uses the user’s manual growth rate and a default or overridden volatility.
Mode selection rules
  • Single Ticker Mode: Ticker has a valid symbol, Portfolio is empty.
  • Portfolio Mode: Portfolio has 2+ tickers with weights, Ticker is empty.
  • Synthetic Mode: Both fields empty, or real‑market fetch fails. Falls back to manual growth.
Design intent
The user should always feel that the mode is intentional, visible, and aligned with how they think about their input: “one asset”, “a portfolio”, or “just assume a growth rate.”
Quick Reference Mode & behavior at a glance

Input expectations
  • Ticker: One symbol, 1–5 letters (e.g., VTI, BND, AAPL). Portfolio must be empty.
  • Portfolio: Tickers with weights, separated by commas: VTI:60, BND:40. Ticker must be empty.
  • Manual growth: Used only when no real‑market configuration is active or data is unavailable.
Good defaults
  • Volatility: 0.15 if no real‑market volatility is available and no override is set.
  • Tax: Auto‑tax uses real‑market growth when available; otherwise, manual growth.
02 · Inputs & Guardrails How user input is interpreted
Ticker field (single asset)
  • Purpose: Model a single asset using real‑market return and volatility.
  • Format: 1–5 letters, e.g., VTI, BND, AAPL.
  • Rule: Portfolio must be empty for this field to activate Real‑Market (Single Ticker) mode.
Portfolio field (multiple assets)
  • Purpose: Model a weighted portfolio using real‑market return and volatility.
  • Format: TICKER:WEIGHT pairs, separated by commas: VTI:60, BND:40.
  • Rule: Ticker must be empty for this field to activate Real‑Market (Portfolio) mode.
Guardrail behaviors
  • Both fields filled: The tool blocks execution and prompts: “Please choose either a single ticker OR a portfolio, not both.”
  • Single ticker in portfolio: If the portfolio parses to exactly one ticker, the tool blocks execution and prompts: “It looks like you entered a single ticker in the portfolio box. Use the Ticker field instead.”
  • No letters in portfolio: Blocks execution and prompts: “Portfolio must contain tickers with weights, like VTI:60, BND:40.”
  • Missing weights: If the user enters VTI, BND or VTI:, the tool prompts: “Each portfolio ticker needs a weight, like VTI:60.”
  • Invalid ticker: If the ticker is not 1–5 letters, the tool prompts: “That doesn’t look like a valid ticker symbol.”
Why guardrails matter
These checks prevent users from accidentally entering the wrong mode, feeding malformed portfolios into the engine, or getting confusing results from invalid tickers.
Guardrail snippet (conceptual)
// After reading and sanitizing inputs
const ticker = $("ticker").value.trim().toUpperCase();

if (ticker !== "" && portfolioStr !== "") {
  alert("Please choose either a single ticker OR a portfolio, not both.");
  return;
}

if (portfolioStr !== "" && !/[A-Za-z]/.test(portfolioStr)) {
  alert("Portfolio must contain tickers with weights, like VTI:60, BND:40.");
  return;
}

if (portfolioStr !== "" && portfolioStr.includes(",") && !portfolioStr.includes(":")) {
  alert("Each portfolio ticker needs a weight, like VTI:60.");
  return;
}

if (portfolioStr !== "") {
  const { tickers } = parsePortfolio(portfolioStr);
  if (tickers.length === 1) {
    alert("It looks like you entered a single ticker in the portfolio box. Use the Ticker field instead.");
    return;
  }
}

if (ticker !== "" && !/^[A-Za-z]{1,5}$/.test(ticker)) {
  alert("That doesn’t look like a valid ticker symbol.");
  return;
}
03 · Modes & Outputs What the user sees and why
Mode behavior
  • Real‑Market (Single Ticker): Uses historical prices for the ticker to compute CAGR, volatility, and Monte Carlo distributions. Mode badge: “Real‑Market (Single Ticker)”.
  • Real‑Market (Portfolio): Uses weighted historical data for all tickers, including correlations, to compute portfolio CAGR and volatility. Mode badge: “Real‑Market (Portfolio)”.
  • Synthetic (Manual Growth): Uses the user’s growth rate and a default or overridden volatility. Mode badge: “Synthetic (Manual Growth)”.
Negative return warning
When expected return is negative
If the real‑market CAGR is negative (e.g., for certain bond funds over specific windows), the tool should surface a subtle warning such as: “This asset has had a negative long‑term return. Results may look unusual.” This helps users interpret outcomes where Traditional may outperform Roth simply because the asset itself has struggled.
Output interpretation
  • Assumed growth rate: Shows the effective CAGR used in the projection (real‑market or manual).
  • Roth vs Traditional totals: Final projected balances after contributions, growth, and taxes.
  • Monte Carlo percentiles: P10, P50, P90 for both Roth and Traditional, based on the volatility and mode.
  • Tax context: Shows current vs retirement tax assumptions, RMDs, and taxable income at retirement.
Debug / power‑user view (optional)
A collapsible “Debug” panel can expose parsed portfolio, mode, expectedReturn source, volatility source, and raw Monte Carlo parameters. This is useful for advisors, power users, and developers validating behavior against expectations.
