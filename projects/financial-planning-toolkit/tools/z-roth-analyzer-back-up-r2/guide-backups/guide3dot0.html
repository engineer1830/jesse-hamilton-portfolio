Service Guide
Roth vs Traditional – Real‑Market Engine
This guide explains how the Roth vs Traditional Analyzer works under the hood — including inputs, modes, glidepath behavior, deterministic projections, RMD logic, tax modeling, and the advisor‑grade insights generated from the simulation. It is designed to give users and maintainers a clear mental model of how the engine behaves and why.
01 · Overview What this tool does
Purpose
The tool compares Roth vs Traditional retirement outcomes using real‑market data, lifecycle glidepath modeling, deterministic projections, and tax‑aware withdrawal logic. It produces a clear, advisor‑grade view of how contributions, growth, taxes, RMDs, and withdrawals interact across a full retirement timeline.
High‑level behavior
  • Real‑market returns: Uses historical CAGR and volatility for a single ticker or weighted portfolio.
  • Lifecycle glidepath (default ON): Automatically adjusts stock/bond allocation each year based on age and retirement horizon, producing more realistic expected returns.
  • Deterministic projection: Runs a year‑by‑year simulation from current age through life expectancy, applying growth, contributions, withdrawals, RMDs, and tax drag.
  • Auto‑tax estimation (default ON): Computes retirement tax rate using RMDs, Social Security, taxable SS, and filing status to avoid unrealistic manual assumptions.
  • Contribution direction toggle: Users can direct future contributions to Roth, Traditional, or a custom split.
  • Advisor‑grade insights: Includes 4% and 5% withdrawal sustainability tests, depletion age, and long‑term survivability scoring.
Design intent
The tool is built to mirror how real advisors model retirement outcomes — blending real‑market data, glidepath logic, tax‑aware withdrawals, and RMD rules into a single, intuitive projection. Users should feel confident that the engine behaves the way a professional planning tool would.

03 · Modes How the engine chooses its behavior
Purpose
The tool supports three distinct modeling modes. Each mode determines how expected return, volatility, and tax behavior are derived. The mode is selected automatically based on user input, ensuring the engine always behaves in a way that matches the user’s intent.
Mode definitions
  • Real‑Market (Single Ticker): Activated when the Ticker field contains a valid symbol and the Portfolio field is empty. The tool fetches historical CAGR and volatility for that asset and uses them in the projection.
  • Real‑Market (Portfolio): Activated when the Portfolio field contains 2+ tickers with weights and the Ticker field is empty. The tool computes weighted CAGR, weighted volatility, and correlation‑adjusted portfolio behavior.
  • Synthetic (Manual Growth): Activated when no real‑market configuration is provided or when data cannot be retrieved. The tool uses the user’s manual growth rate and a default or overridden volatility.
Mode selection rules
  • Single Ticker Mode: Ticker has a valid symbol, Portfolio is empty.
  • Portfolio Mode: Portfolio contains 2+ tickers with weights, Ticker is empty.
  • Synthetic Mode: Both fields empty, or real‑market fetch fails.
Mode badge (UI)
The mode badge updates in real time as the user types. This ensures the user always knows which modeling approach is active.

Why modes exist
Each mode represents a different level of realism and data availability. Real‑market modes use historical behavior to anchor projections, while Synthetic mode provides a fallback when users prefer or require a manual assumption. The engine always chooses the most realistic mode available.
04 · Deterministic Engine How the projection is calculated
Purpose
The deterministic engine runs a year‑by‑year simulation from the user’s current age through retirement and into late life. Each year applies growth, contributions, withdrawals, RMDs, Social Security, and tax drag. This produces a clear, stable projection that forms the basis of the chart and the Roth vs Traditional comparison.
Yearly loop
Each simulation year follows a consistent sequence of steps. This ensures the projection behaves predictably and mirrors real retirement planning workflows.
  • 1. Determine expected return: Based on the active mode (real‑market or synthetic) and glidepath allocation for that year.
  • 2. Apply contributions: Contributions are added to Roth, Traditional, or both depending on the user’s contribution direction settings.
  • 3. Apply growth: Both Roth and Traditional balances grow at the expected return for that year.
  • 4. Apply withdrawals (if retired): If the user is past retirement age, withdrawals are taken from Traditional first (to satisfy RMDs), then Roth if needed.
  • 5. Apply RMDs: Starting at age 73, the IRS divisor is applied to Traditional balances. RMDs count as taxable income and reduce the Traditional balance accordingly.
  • 6. Apply tax drag: Taxes are applied to Traditional withdrawals and RMDs using either the user’s manual retirement tax rate or the auto‑tax estimate.
  • 7. Add Social Security (if applicable): Social Security income is added at the user’s claim age, with taxable SS rules applied based on filing status and other income.
  • 8. Compute ending balances: After growth, contributions, withdrawals, RMDs, and taxes, the year’s ending balances become the starting balances for the next year.
Growth logic
Growth is applied once per year using the expected return for that year. When glidepath is enabled, expected return changes over time as the allocation shifts from aggressive to conservative.

Withdrawal logic
Withdrawals begin at retirement age. The tool follows a tax‑efficient withdrawal order:
  • 1. RMDs first: Required minimum distributions must be taken from Traditional.
  • 2. Additional need from Traditional: If the user needs more than the RMD, Traditional is tapped next.
  • 3. Roth last: Roth is used only when Traditional cannot cover the need.
This mirrors common advisor practice: preserve Roth for as long as possible due to its tax‑free growth and tax‑free withdrawals.
Why deterministic?
Deterministic projections provide a stable, easy‑to‑interpret baseline for comparing Roth and Traditional outcomes. They avoid the noise of Monte Carlo randomness while still incorporating realistic return assumptions through real‑market data and glidepath logic.
05 · Lifecycle Glidepath Age‑based allocation model
Purpose
The lifecycle glidepath models how real investors typically shift from growth to preservation as they age. When enabled, the tool automatically adjusts the stock/bond allocation each year based on the user’s age and retirement horizon. This produces more realistic expected returns than a fixed growth rate.
Allocation phases
The glidepath is divided into four intuitive phases. Each phase represents a typical risk posture for that stage of life.
  • Aggressive: 100% stock until 10 years before retirement. Maximizes long‑term growth.
  • Moderate: 65% stock / 35% bonds from 10 to 2 years before retirement. Reduces volatility while maintaining growth potential.
  • Preserve: 50% stock / 50% bonds from 2 years before retirement until age 70. Balances stability with inflation protection.
  • Legacy: 35% stock / 65% bonds at age 70 and beyond. Prioritizes capital preservation and income stability.
Expected return logic
Each year of the simulation uses a blended expected return based on that year’s stock/bond allocation. Stock returns come from the active mode (real‑market or synthetic), while bond returns use historical bond market behavior.

Volatility logic
Volatility is also blended based on the year’s allocation. This affects Monte Carlo simulations (if enabled) and helps users understand the risk profile of each phase.

  • Higher bond allocation → lower volatility.
Integration with deterministic engine
The glidepath directly influences the deterministic projection by providing the expected return and volatility for each year. This ensures the projection reflects realistic age‑based risk changes rather than a flat growth assumption.
Deterministic charts use the first‑year glidepath return for clarity, while Monte Carlo simulations (if enabled) use year‑specific volatility.
Why glidepath is the default
Glidepath modeling mirrors the behavior of target‑date funds and professional retirement planning tools. It provides a more realistic projection of long‑term outcomes and helps users understand how risk naturally evolves over time. For these reasons, glidepath is enabled by default.
06 · RMD Engine Required Minimum Distribution logic
Purpose
Required Minimum Distributions (RMDs) are mandatory withdrawals from Traditional accounts beginning at age 73. They are a major driver of retirement taxes and a key differentiator between Roth and Traditional outcomes. The RMD engine ensures the projection reflects real IRS rules and their impact on long‑term balances.
IRS divisor
The tool uses the official IRS Uniform Lifetime Table to compute RMDs. Each year has a corresponding divisor based on age. The RMD is calculated as:
RMD = Traditional Balance ÷ IRS Divisor
As the divisor decreases with age, RMDs grow larger over time — even if the balance stays the same.
Gross vs net withdrawals
RMDs are treated as gross withdrawals. Taxes are applied after the RMD is taken, reducing the net amount available for spending.
  • Gross RMD reduces Traditional balance.
  • Taxes are applied to the RMD amount using the retirement tax rate.
  • Net RMD contributes to meeting the user’s withdrawal need.
Interaction with withdrawals
RMDs are always taken first. If the user’s annual withdrawal need exceeds the RMD, the additional amount is taken from Traditional (if available) and then Roth.
This ordering mirrors tax‑efficient withdrawal strategies used by advisors: satisfy RMDs, then draw from Traditional, and preserve Roth for last.
Tax implications
RMDs increase taxable income, which can:
  • Increase the user’s effective retirement tax rate.
  • Cause more Social Security to become taxable.
  • Accelerate depletion of Traditional accounts.
These effects are automatically captured when Auto‑Tax is enabled.
Impact on Roth vs Traditional comparison
RMDs are one of the biggest drivers of long‑term differences between Roth and Traditional outcomes. Because Roth accounts have no RMDs, they:
  • Grow tax‑free for life.
  • Do not force taxable income in retirement.
  • Preserve more wealth for late‑life and legacy phases.
The RMD engine is essential for producing a fair, realistic comparison between Roth and Traditional strategies — especially for users with large Traditional balances.
07 · Phase Shading Visualizing risk posture over time
Purpose
Phase shading provides a visual overlay on the projection chart that reflects the user’s risk posture at each stage of life. It helps users understand how their allocation — and therefore their expected return and volatility — evolves over time.
The four phases
The shading corresponds directly to the glidepath phases. Each phase has a distinct color and represents a meaningful shift in investment strategy.
  • Aggressive: 100% stock allocation. High growth potential and high volatility. Typically spans early career through mid‑career.
  • Moderate: 65% stock / 35% bonds. Balances growth and stability as retirement approaches.
  • Preserve: 50% stock / 50% bonds. Reduces volatility during the transition into retirement.
  • Legacy: 35% stock / 65% bonds. Prioritizes capital preservation and income stability in late retirement.
How shading is computed
The shading engine evaluates the user’s age each year and assigns the corresponding glidepath phase. The chart then renders a background color for that year’s segment.
  • Each year is mapped to a glidepath phase.
  • Each phase has a unique color with subtle transparency.
  • Shading spans the full width of the chart behind the projection line.
Shading is purely informational — it does not affect the projection itself. It simply visualizes the glidepath phases already used in the deterministic engine.
Why shading helps
Phase shading helps users intuitively understand how their risk posture changes over time. It highlights the transition from aggressive growth to preservation and legacy phases, making the projection easier to interpret at a glance.
08 · Outputs What the tool produces and how to interpret it
Purpose
The output section summarizes the results of the projection and highlights the key differences between Roth and Traditional strategies. It includes final balances, tax‑aware comparisons, sustainability insights, and optional Monte Carlo percentiles.
Comparison summary
The comparison summary provides a high‑level view of which strategy performs better under the modeled assumptions. It includes:
  • Final Roth balance
  • Final Traditional balance (after taxes)
  • Total taxes paid in retirement
  • Net after‑tax wealth at end of plan
The summary is tax‑aware and incorporates RMDs, taxable Social Security, and filing status.
Retirement tax estimate
When Auto‑Tax is enabled, the tool computes an estimated retirement tax rate based on:
  • RMD income
  • Social Security income
  • Taxable Social Security rules
  • Filing status
This estimate is used to compute after‑tax Traditional withdrawals and final balances. Emphasis on estimate.
Break‑even tax rate
The break‑even tax rate is the retirement tax rate at which Roth and Traditional strategies produce the same after‑tax outcome. It helps users understand how sensitive the comparison is to future tax assumptions.

09 · Pro Insights Advisor‑grade interpretation of long‑term outcomes
Purpose
Pro Insights translate the raw projection into meaningful, advisor‑grade guidance. These metrics help users understand not just how much money they may have, but how sustainable their retirement plan is under realistic withdrawal assumptions.
4% withdrawal analysis
The 4% rule is a classic retirement guideline suggesting that withdrawing 4% of the initial portfolio each year (adjusted for inflation) is historically sustainable for a 30‑year retirement. The tool evaluates whether the user’s projected balances support this rule.
  • Determines if 4% withdrawals deplete the portfolio before life expectancy.
  • Highlights whether Roth or Traditional provides better sustainability.
A successful 4% test indicates strong long‑term sustainability under conservative assumptions.
5% withdrawal analysis
The 5% test provides a more aggressive scenario. It helps users understand how sensitive their plan is to higher spending needs or unexpected expenses. The key consideration is how much is needed and what can be sustained. There is not a one-size fits all answer.
  • Tests whether the portfolio can sustain a 5% withdrawal rate.
  • Shows how quickly balances decline under higher spending.
Passing the 5% test is rare — failing it is normal and not a sign of poor planning. There are some who advocate for 5% as well.
Sustainability score
The sustainability score summarizes how resilient the user’s retirement plan is under realistic withdrawal assumptions. It incorporates:
  • 4% test result
  • 5% test result
  • Depletion age
  • RMD impact
A higher sustainability score indicates a more robust retirement plan with greater margin for unexpected expenses or market volatility.
Depletion age
Depletion age is the age at which the portfolio reaches zero under the modeled withdrawal assumptions. It is one of the most intuitive and impactful metrics for users.
  • Later depletion age = stronger plan.
  • Earlier depletion age = higher risk of shortfall.
Depletion age is especially useful when comparing Roth vs Traditional outcomes.
Behavioral interpretation
Pro Insights help users understand the “why” behind the numbers. They highlight how taxes, RMDs, glidepath behavior, and withdrawal rates interact to shape long‑term outcomes. It is typical that the life-style lived during working years will mirror the life-style of retirement. Many discover that spending is more early in retirement as health permits.
These insights turn the projection from a simple chart into a meaningful retirement planning conversation.
How advisors use these insights
Advisors use sustainability tests, depletion age, and tax‑aware comparisons to guide contribution strategy, withdrawal sequencing, and long‑term planning. These insights help users make informed decisions that align with professional best practices.
10 · Guardrails Ensuring valid inputs and meaningful results
Purpose
Guardrails prevent invalid, ambiguous, or contradictory configurations from entering the engine. They ensure the tool always runs in a well‑defined mode and produces meaningful, trustworthy results. These rules protect both the user experience and the integrity of the projection.
Input validation
The tool validates all user inputs before running the projection. This includes checking for missing fields, invalid formats, and contradictory configurations.
  • Age must be between 18 and 100.
  • Retirement age must be greater than current age.
  • Balances and contributions must be non‑negative.
Mode enforcement
Guardrails ensure the engine always selects a single, unambiguous mode. This prevents situations where the tool might otherwise misinterpret the user’s intent.
  • Single Ticker Mode: Only the Ticker field is filled.
  • Portfolio Mode: Only the Portfolio field is filled with 2+ tickers.
  • Synthetic Mode: Neither field is filled, or real‑market data fails.
If both Ticker and Portfolio are filled, the tool blocks the run and prompts the user to choose one mode.
A single‑ticker portfolio is not allowed — users must use the Ticker field instead.
Portfolio parsing
Portfolio entries must follow a strict format to ensure accurate parsing and weighting. Guardrails prevent malformed portfolios from entering the engine.
  • Format: TICKER:WEIGHT (e.g., VTI:60).
  • Weights must sum to 100%.
  • At least 2 tickers are required for Portfolio Mode.
Ticker rules
Ticker symbols must follow standard market conventions. Guardrails ensure the tool only attempts to fetch data for valid symbols.
  • 1–5 uppercase letters only (e.g., VTI, QQQ).
  • No special characters or numbers.
  • If the symbol fails lookup, the tool falls back to Synthetic Mode.
A single‑ticker portfolio is not allowed — users must use the Ticker field instead.
Why guardrails matter
Guardrails ensure the engine always runs in a clean, predictable state. They prevent ambiguous inputs, protect users from accidental mistakes, and guarantee that the projection reflects the user’s true intent. This is essential for producing advisor‑grade results.
